## 2. 生成质量评测

### 2.1 LLM-as-judge 评估（50题，2026-07-18）

| 指标 | 得分 |
|------|------|
| **生成正确率** | **78%** (53/68) |
| 平均置信度 | 0.99 |
| 评估方式 | DeepSeek judge（语义判断，非子串匹配） |

> 注：子串匹配评估仅 56%（38/68），LLM-as-judge 评估 78%，差了 22 个百分点——说明 DeepSeek 的回答语义正确但表述不同，子串匹配过于严格。

### 2.2 典型失败案例分析

| 问题 | 结果 | 原因 |
|------|------|------|
| 绩效分布饼图S级占比 | ❌ | 图表 OCR 文本不完整 |
| 晋升需要绩效达到什么标准 | ❌ | chunk 中信息隐含，LLM 未提取 |
| 辞职要提前多久 | ❌ | chunk 中写"30天"，LLM 答"提前一个月"语义等价但 judge 判否 |
| 机密数据禁止通过什么传输 | ❌ | 合规条款表述特殊 |

### 2.3 检索→生成全链路

| 阶段 | 指标 | 
|------|------|
| **检索 Hit@3** | **86.5%** |
| ↓ | |
| **生成正确率** | **78%** |
| **链路损耗** | **8.5 个百分点** |

检索和生成之间的 8.5 点损耗主要来自：LLM 对隐含信息的提取失败、表述方式的差异、以及图表 OCR 的不完整。

## 5. 结论

1. **中文知识检索优秀**：Golden QA 员工手册 21/25 章节 100% 命中，整体 Hit@3=86.5%
2. **英文通用知识**：CRAG 检索受限于 snippet 摘要的信息量，需完整 HTML 页面才能提升
3. **生成系统**能正确引用来源，无上下文时正确拒答
4. **限流系统**在生产配置下正确工作
5. **性能**：读操作 P50 < 30ms，检索（含嵌入）P50 ≈ 1.8s

### 改进建议

- 图表类（chart/png）80% 可考虑优化 OCR 管道
- CRAG 全量评测需更大内存环境（>2GB）
- 检索延迟 1.8s 主要是通义嵌入 API 耗时，本地缓存可加速重复 query
