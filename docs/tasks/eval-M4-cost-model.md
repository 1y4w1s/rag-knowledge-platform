## 0. 一句话（大白话）

知岸花钱的地方就三块：**上传时把文档切成块并向量化（通义 embed）**、**对话时把问题向量化 + 精排（通义 embed + rerank）**、**有依据时让 DeepSeek 写回答**。列表翻页、登录、Dashboard 统计 **不调外部 AI**，不花 API 钱（M2 已证：慢在数据库，不在 token）。

## 2. 单次对话成本（M4-2）

### 2.1 数据流（对应代码）

```
用户提问
  → E2 问题嵌入 1 次（embed_texts([query])）
  → 库内 hybrid 召回 Top-20（本地 Postgres，不计费）
  → R1 rerank 1 次（20 条候选 → Top-5）     [RERANK_ENABLED=false 时跳过]
  → C1 DeepSeek 流式生成                      [无片段时走 C2，¥0]
```

### 2.2 Token 估算约定

| 符号 | 含义 | 默认值（demo 估算用） |
|------|------|----------------------|
| `T_q` | 用户问题 token | **50**（约 50 字中文问句） |
| `T_sys` | 系统提示词 token | **200** |
| `K` | 喂给 LLM 的片段数 | **5**（`LLM_TOP_K`） |
| `T_chunk` | 单片段正文 token | **600**（切片上限 1000 字，实际喂入含标题路径；TECH 总上限 ≤3000 token） |
| `N_rerank` | rerank 候选条数 | **20**（`RERANK_INPUT_TOP_N`） |
| `T_rerank_doc` | rerank 单条文档 token | **800**（`parent_content` 优先，常比 excerpt 长） |
| `T_out` | 模型回答 token | **500**（plan 假设 ~500 字回答） |

**中文粗算口诀**：1 个汉字 ≈ **1 token**（略保守，方便心算）。

### 2.3 公式

**有依据、rerank 开启时：**

```
T_embed  = T_q
T_rerank = T_q + N_rerank × T_rerank_doc
T_in_llm = T_sys + K × T_chunk + T_q + 固定格式开销(~100)
T_out_llm = T_out

费用_embed  = (T_embed / 1000) × 0.0007
费用_rerank = (T_rerank / 1000) × 0.0005
费用_chat   = (T_in_llm / 1_000_000) × 1.0 + (T_out_llm / 1_000_000) × 2.0

单次对话 ≈ 费用_embed + 费用_rerank + 费用_chat
```

**无依据拒答（R4-2）**：`费用 ≈ 费用_embed + 费用_rerank`（仍检索；不调 DeepSeek）

**rerank 关闭**（`RERANK_ENABLED=false`）：去掉 `费用_rerank`

### 2.4 代入示例（默认参数）

| 步骤 | Token | 计算 | 费用 |
|------|-------|------|------|
| E2 问题嵌入 | 50 | 0.05 × 0.0007 | **¥0.000035** |
| R1 rerank | 50 + 20×800 = 16,050 | 16.05 × 0.0005 | **¥0.0080** |
| C1 DeepSeek 输入 | 200 + 5×600 + 50 + 100 = 3,350 | 3,350 / 1e6 × 1 | **¥0.0034** |
| C1 DeepSeek 输出 | 500 | 500 / 1e6 × 2 | **¥0.0010** |
| **合计（典型有依据）** | — | — | **≈ ¥0.012 / 次** |

**口播版**：「一次带引用的对话，大约 **1～2 分钱**；一天聊 50 次也就 **几毛钱到一块多**。」

## 3. 单次上传成本（M4-3）

### 3.1 数据流

```
上传文件
  → 本地解析 + 结构切片（pdfplumber / **扫描件 OCR**，见 §7）
  → 过滤掉 chunk_kind=parent，只对可检索块嵌入
  → E1 一次 API 调用可 batch 多条文本（pipeline 整文档一次 embed_texts）
  → 写入 Postgres/pgvector（不计费）
```

### 3.2 切片数量公式

来自 `IngestionConfig`：`max_chars=1000`，`overlap_max_chars=150`。

```
有效块长 L_eff ≈ max_chars - overlap ≈ 850 汉字
N_chunk ≈ ceil(文档总字数 / L_eff)
```

**按页粗估**（员工手册类 PDF）：

```
每页字数 P_page ≈ 500 汉字（400～700 取中）
N_chunk ≈ ceil(页数 × P_page / 850)
```

**嵌入 token（单块）**：

```
T_embed_chunk ≈ len([heading_path] + content) ≈ 30 + min(字数, 1000) ≈ 1030
T_embed_doc   = N_chunk × T_embed_chunk
费用_upload   = (T_embed_doc / 1000) × 0.0007
```

> `parent` 块 **不嵌**、不计费；表格块单独成块，页数少但块可能更长。

### 3.3 例子

#### 例 A · 10 页 PDF 员工手册

| 项 | 值 |
|----|-----|
| 总字数 | 10 × 500 = **5,000** 字 |
| 块数 | ceil(5000/850) = **6** 块 |
| 嵌入 token | 6 × 1030 ≈ **6,180** |
| **费用** | 6.18 × 0.0007 ≈ **¥0.004**（半分钱级） |

#### 例 B · 80 页制度汇编

| 项 | 值 |
|----|-----|
| 总字数 | 80 × 500 = **40,000** 字 |
| 块数 | ceil(40000/850) = **48** 块 |
| 嵌入 token | 48 × 1030 ≈ **49,440** |
| **费用** | 49.44 × 0.0007 ≈ **¥0.035**（约 3～4 分钱） |

#### 例 C · 重试 / 重嵌入

| 场景 | 费用 |
|------|------|
| 上传失败点「重试」 | 与首次相同（删旧块后重跑管道） |
| 换模型后 `re_embed_all.py` | **每个可检索 chunk 再付一遍 E1** |

### 3.4 API 调用次数（便于对账）

| 场景 | embed HTTP 次数 | 说明 |
|------|-----------------|------|
| 上传 1 个文档 | **1 次** | 整文档所有块一次 batch |
| 对话 1 次 | **1 次** | 仅 query |
| 重嵌入 | **ceil(块数 / 25)** | `RE_EMBED_BATCH_SIZE=25` |

### 4.3 建议预算线（给你拍板用）

| 档位 | 嵌入月上限 | 对话月上限 | 合计告警线 | 适用 |
|------|-----------|-----------|-----------|------|

| 做法 | 省什么 | 代价 |
|------|--------|------|

## 6. 面试 30 秒口播

> 我们把 API 成本拆成嵌入、精排、生成三层。上传按切片字数走通义 embed，一份十页手册大概半分钱；对话每次先嵌问题、再对 20 条候选 rerank，最后 DeepSeek 吃 Top-5 片段生成，带引用大约一分钱量级。Demo 月我们设一百块告警，正常答辩准备实际个位数到十几块；列表压测用 S/L 档元数据 seed 不嵌向量，避免万库场景把嵌入费打爆。

## 7. OCR 算力成本（F4 · 本地 CPU · 无 API token）

> **一句话**：扫描 PDF 的「认字」在**你自己的服务器 CPU** 上跑 PaddleOCR，**不向通义/DeepSeek 按 token 付钱**；OCR 完成后嵌入/对话仍走 §1 的 E1/C1 计费。

### 7.1 花什么、不花什么

| 项 | 是否花 API 钱 | 说明 |
|----|---------------|------|
| PaddleOCR 认字 | **¥0** | 开源本地引擎；首次运行会下载模型（约 **100～200 MB** 磁盘） |
| poppler 渲染 PDF 页 | **¥0** | 系统包，把 PDF 页变成图给 OCR |
| OCR 后的 embed / 对话 | **同 §1～§3** | OCR 只多产出文本块；**嵌入 token 与文字层 PDF 同公式** |
| 多模态 vision LLM | **不做（F5）** | 若改用「让 LLM 看图」，会按图/token 计费 — **F4 明确不走这条路** |

### 7.2 算力粗估（2G 小机 · 答辩量级）

| 假设 | 粗算 | 口播 |
|------|------|------|
| 单页 OCR | **约 3～15 秒/页**（视 CPU、清晰度） | 「十页扫描件整理可能要一两分钟，列表会显示整理中」 |
| 单文件上限 | 默认 **30 页**（`OCR_MAX_PAGES`） | 超限直接 failed，避免整本扫描册拖死小机 |
| 内存 | OCR 时 API 进程 **短暂升高**（模型 + 页图） | 与 §4.3.7「10 页一批入库」叠加时更吃内存 |
| 并发 | 多个用户同时传扫描件 | BackgroundTasks **串行感明显**；demo 可接受，生产须限流/队列（未做） |

**Demo 月 OCR 算力**：答辩期若上传 **≤10 份 × 平均 5 页扫描件**，CPU 时间 **约 tens of minutes 总量**，仍在 2G 单机可承受范围；**不必**为 OCR 单独设 API 预算线。

### 7.3 省钱 / 关断

| 做法 | 效果 |
|------|------|
| `OCR_ENABLED=0` | 扫描件恢复「不支持扫描件」；**零 OCR CPU** |
| 不装 `requirements-ocr.txt` | CI / 轻量环境默认行为；扫描件 failed +「OCR 服务未启用」 |
| 控制扫描上传份数 | 线性减少 CPU；嵌入费仍按 §3 计 |
| 勿一次传 &gt;30 页扫描册 | 避免长时间 processing + OOM 风险 |

### 7.4 你怎么验

- [ ] 能区分：**OCR 不占通义/DeepSeek token** · 嵌入仍占  
- [ ] 能口播：**扫描件认字在本地 CPU，一份十页扫描 PDF 整理可能 1～3 分钟**  
- [ ] 知道 **`OCR_ENABLED=0`** = 关 OCR、省 CPU，扫描件仍失败  
- [ ] 部署须 **poppler + optional pip OCR 依赖**（见 `DEPLOY.md` §4.1 · `requirements-ocr.txt`）
