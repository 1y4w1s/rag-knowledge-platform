## WS-1-1 核心定义

| 概念 | 定义 |
|------|------|
| **账号** | 一个用户（邮箱/用户名），注册即拥有 **「我的空间」** |
| **我的空间** | 个人资料库（`owner_user_id`）；永远存在 |
| **团队空间** | 组织资料库（`owner_org_id`）；有 `organization_members` 记录才可进入 |
| **当前工作区** | 用户在前端切换的上下文；列表/Dashboard/对话 **只展示当前空间** |
| **加入团队** | 增加成员关系（邀请码 / 设置填码 / 管理员加邮箱）；**不**用 flip `account_type` 切换整站 |
| **离开团队** | 成员自退或管理员移除；回到「我的空间」；个人库始终保留 |

**MVP 约束**：一人至多 **一个团队**（与现后端一致）。

## WS-1-2 注册（三步 + 可返回） ✅ 已确认（2026-07-05 · 组织名软引导 + 展示名语义）

| 步 | 内容 |
|----|------|
| **1** | 个人 / 团队 |
| **2** | 个人 → 跳过；团队 → **创建者**（组织名 *）或 **成员**（邀请码 *） |
| **3** | 用户名、邮箱、密码、确认密码、昵称可选 → **创建账号** |

| 选择 | 注册结果 | 默认工作区 |
|------|----------|------------|
| 个人 | 仅我的空间 | 我的空间 |
| 团队·创建者 | 我的空间 + 新团队 + **Owner**（`org_role=admin` 且 `is_owner=true`） | 团队空间 |
| 团队·成员 | 我的空间 + member（码校验） | 团队空间 |

**规则**

- **B1**：邀请码无效/过期 → 注册失败，不创建账号。
- **I1（2026-07-05 ✅）**：邀请码 **可多人用**；无「已被使用」提示；无效/过期/已撤销统一文案。
- **I2（2026-07-05 ✅）**：团队·成员 · 第 2 步点「继续」**预校验**邀请码；失败留本步；成功返回团队展示名供第 3 步 chip。
- **团队·成员**：邀请码 **必填**；无码说明引导 **个人注册**，登录后在 **账号设置** 填码或由管理员加邮箱（W3）。
- 底部「已有账号？注册」小字 **不恢复**（靠顶 Tab）。
- 步骤条（4A）：个人 2 步（用法→登录信息）；团队 3 步（用法→团队角色→登录信息）。

### 2.1 团队显示名称（团队 · 创建者 · 第 2 步）

**原则**：注册页 **不**硬 cap 低于 255；用户填的是 **壳层展示名**（见 **§2.4**），不是「只能存简称」。长名靠 **壳层展示算法 + 组织设置** 消化，不靠注册挡合规全称。

| 项 | 定稿 |
|----|------|
| 字段标签 | **「团队显示名称」**（Implement 统一此文案；**不再**用「组织名称 *」） |
| Placeholder | 例如 **知岸科技** |
| Helper 文案 | 用于侧栏切换器展示，**建议 20 字以内**；若需存更长名称，注册后可在 **组织设置**（WS-2-9）修改 |
| **硬校验**（不变） | trim 后 **1～255** 非空；空或超长 → **阻止提交** |
| **软校验 · >40 字** | 输入框下方 **黄色提示**（可继续填、可点下一步） |
| **软校验 · >80 字** | 点「下一步」或「创建账号」前弹 **确认对话框**：「名称较长，侧栏可能显示不全；可在组织设置改为简称」；用户确认后继续 |
| 后端/API | 仍接受 1～255；写入 `organizations.name`（**展示名**，§2.4） |

### 2.2 验收（组织名 · 注册）

| # | 操作 | 预期 |
|---|------|------|
| R1 | 创建者 · 输入「知岸科技」（≤20 字） | 无警告；注册成功 |
| R2 | 创建者 · 输入 45 字名称 · 点下一步 | 见 **黄色提示**；仍可进入第 3 步 |
| R3 | 创建者 · 输入 90 字名称 · 提交 | 先见确认框「名称较长，概览可能显示不全」；确认后注册成功 |
| R4 | 创建者 · trim 为空 | **硬拦**；不创建账号 |
| R5 | 创建者 · 256 字 | **硬拦**（255 上限） |
| R6 | R3 注册成功 · 登录团队 Dashboard | 侧栏 **策略 C** + Popover 全称（WS-2-1 **§1.1.2**）；badge **仅**「N 名成员 ›」 |
| R7 | R3 且名称 >80 字 · 首次进团队 Dashboard | 见 **Info Banner**「名称较长，建议在组织设置改为简称」（§2.5）；dismiss 后不再强弹 |
| R8 | 创建者 · 输入 6 字「知岸科技」 | 侧栏团队段 **完整显示**；无 Popover 必填（仍可点看全称） |

### 2.4 组织名 · 数据语义（根治 · MVP vs Phase 2）

**这节定什么**：注册填的名字在库里算什么；和壳层展示怎么对齐。

| 项 | MVP（W1～W2） | Phase 2（政企 / 合规） |
|----|---------------|------------------------|
| **`organizations.name`** | **团队展示名**（shell 唯一来源；1～255） | 保留为 **display_name** 语义 |
| **`legal_name`** | **无**独立字段；展示名即用户填写的唯一名称 | 可选 **VARCHAR(255)**；发票/合同用；**不**写入侧栏 seg |
| 注册写入 | 创建者填的值 → `name` | 可拆：展示名必填 + 法定名可选 |
| 壳层展示 | **只读 `name`**，经 **策略 C**（§1.1.2）+ Popover | seg 读 display；法定名仅组织设置（Phase 2） |
| 组织设置 | Admin 可 **改 `name`（展示名）** 1～255 | 展示名 + 法定名 **分行编辑** |

**不做（MVP）**：为根治长名在注册硬限 32 字；为 MVP 加 migration 拆双字段（Phase 2 单独立项）。

### 2.5 注册后 · 长名引导（根治 · 软限制闭环）

### 2.3 乱操作 / 边界（组织名 · 注册）

| # | 乱操作 | 系统怎么处理 | 你怎么验 |
|---|--------|--------------|----------|
| E1 | 粘贴含首尾空格的组织名 | trim 后校验；存库为 trim 后值 | 侧栏无多余空格 |
| E2 | 仅空格 / 全角空格 | 硬拦：非空校验失败 | 无法下一步 |

| E6 | 注册 >80 字 · Banner dismiss 后刷新 | Banner 仍隐藏（dismiss 持久） | 侧栏 Popover 仍可看全称 |

## WS-1-6 团队角色 · Owner + Admin（✅ 已确认 2026-07-05）

**这节定什么**：创建者如何放权；副手能否「干掉」创建者。

### 6.1 三档角色

| 角色 | 来源 | 团队空间能做什么 |
|------|------|------------------|
| **Owner（所有者）** | 注册选「团队·创建者」 | 与 Admin 相同：成员管理、组织设置、建库/删库、上传/删文档、发邀请码 |
| **Admin（管理员）** | Owner **设为管理员** | 同上（日常副手） |
| **Member（成员）** | 邀请码 / 邮箱加入 | 看库、预览、对话（只读） |

**数据**：`organizations.created_by_user_id`（或等价 `organization_members.is_owner`）；Implement 时二选一，须能判断 Owner。

### 6.2 Owner 保护规则

| 规则 | 定稿 |
|------|------|
| 不可被降级 | 其他 Admin **不能**把 Owner 降为 Member |
| 不可被移除 | 其他 Admin **不能**移除 Owner（沿用「不能移除 admin」并加强为 Owner） |
| 组织存续 | 至少保留 **1 名 Owner**（创建者即首任 Owner） |
| 自退 | Owner 离开团队前须 **转让所有权**（W3；MVP 可先不做转让 UI，但 API/规则须预留） |

### 6.3 放权（仅 Owner 可改角色）

| 操作 | 谁可操作 | 对象 |
|------|----------|------|
| 设为管理员 | **Owner** | Member → Admin |
| 降为成员 | **Owner** | Admin（非 Owner）→ Member |
| 移除成员 | Owner **或** Admin | 仅 **Member** |
| 改 Admin 角色 | **仅 Owner** | Admin ↔ Member |

**MVP 简化**：Admin 不能提拔/降级任何人（避免副管互相改权）；仅 Owner 在成员管理页改角色。

### 6.4 侧栏 / API 判断

- 管理菜单（成员管理、组织设置）：`当前工作区=团队 && (org_role===admin)`（Owner 必为 admin）
- 改角色 API：`require_owner` 或 `acting_user.is_owner`
- JWT / `/me` 响应增加 `is_owner: boolean`（前端展示「创建者」标签）

### 6.5 成员管理页展示

| 行 | 角色标签 | 操作（Owner 登录时） |
|----|----------|----------------------|
| 创建者 | **所有者** | 无降级/移除；W3+ 可选「转让所有权」 |
| 副管 | 管理员 | 「降为成员」 |
| 普通成员 | 成员 | 「设为管理员」「移除」 |

Admin（非 Owner）登录：可添加/移除 **Member**，**看不到**改角色按钮（或只读列表）。

### 6.6 验收摘要（W2/W3 · 成员）

| # | 操作 | 预期 |
|---|------|------|
| O1 | Owner 设 Member 为 Admin | 对方出现管理菜单 |
| O2 | Admin 尝试降级 Owner | 403 |
| O3 | Admin 尝试移除 Owner | 403 |
| O4 | Owner 降 Admin 为 Member | 成功；对方管理菜单消失 |
| O5 | 成员列表 | Owner 行显示「所有者」 |
