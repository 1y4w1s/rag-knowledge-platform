# 第一部分：现状审计（六大维度）

## 1.1 视觉设计一致性

### 现状概述

项目采用 CSS 变量驱动的设计令牌系统，以「暖白 + 赤陶橙」为核心色彩语言，避开了企业级产品常见的冷蓝/冷灰。这一方向值得肯定，但在执行层面存在大量一致性问题。

### 具体问题

#### 问题 1：按钮圆角体系混乱（严重）

| Variant | 圆角 | 用途 |
|---------|------|------|
| `default` | `rounded-full`（胶囊形） | 默认操作 |
| `brand` | `rounded-[10px]` | 品牌操作 |
| `auth` | `rounded-[10px]` | 认证页操作 |
| `outline` | `rounded-full`（胶囊形） | 次要操作 |
| `ghost` | `rounded-lg`（8px） | 幽灵操作 |
| `authOutline` | `rounded-[10px]` | 认证页次要 |

**影响**：同一页面内出现三种圆角语言（胶囊 / 10px / 8px），视觉碎片化严重。用户在认知上无法建立「按钮 = 某种形状」的稳定心理模型，增加了视觉扫描成本。特别是 `default`（胶囊黑底）和 `brand`（10px 橙底）两种主操作按钮形状不同，在 Dashboard 同时出现时尤为突兀。

#### 问题 2：字号体系碎片化（严重）

全站使用了至少 **15 种非标准字号值**，缺乏统一的 type scale：

```
text-[0.58rem] → text-[0.62rem] → text-[0.64rem] → text-[0.65rem] →
text-[0.68rem] → text-[0.72rem] → text-[0.75rem] → text-[0.76rem] →
text-[0.78rem] → text-[0.8rem] → text-[0.8125rem] → text-[0.82rem] →
text-[0.875rem] → text-[0.9rem] → text-[0.92rem] → text-[1.05rem] →
text-[1.35rem] → text-[2.25rem]
```

其中 `0.68rem`、`0.72rem`、`0.75rem`、`0.78rem` 之间的差异仅 0.03rem（约 0.5px），肉眼几乎无法区分，但增加了维护成本和认知负担。

**影响**：层级关系模糊——用户无法通过字号快速判断信息重要性。开发者也无法预测「下一个标签该用什么字号」，导致每次开发都在做微观决策。

#### 问题 3：色彩令牌冗余与语义混乱（中等）

存在多个语义重叠的橙色令牌：

| 令牌 | 值 | 实际用途 |
|------|----|---------|
| `--action` | `#cb6b3d` | 主操作色 |
| `--action-hover` | `#b85a2e` | 主操作悬停 |
| `--auth-action` | `#cb6b3d` | 认证页操作（与 --action 完全相同） |
| `--auth-action-hover` | `#b85a2e` | 认证页悬停（与 --action-hover 完全相同） |
| `--nav-active-text` | `#cb6b3d` | 导航活跃文字（与 --action 完全相同） |
| `--acc` | `#6e6560` | 强调色（灰棕，与橙色系完全不同） |

`--auth-action` 和 `--action` 是完全相同的值，却被当作两个独立令牌维护。而 `--acc`（灰棕 `#6e6560`）作为"强调色"与 `--action`（赤陶橙 `#cb6b3d`）语义重叠但色值差异巨大，导致 `.cite-chip-num` 使用 `bg-accent`（灰棕底白字）而 `.approval-card-btn-adopt` 使用 `bg-[var(--action)]`（橙底白字），同为"强调性元素"却呈现截然不同的视觉重量。

**影响**：主题切换时需同步修改多处令牌；新开发者难以判断何时用 `accent` 何时用 `action`。

#### 问题 4：CSS 组件类膨胀（中等）

`index.css` 单文件已达 **1517 行**，`@layer components` 中定义了约 **80+ 个自定义组件类**（`.chat-user-msg`、`.cite-chip`、`.approval-card`、`.thread-list-item` 等）。这些类本质上应该是一棵组件树，却被平铺在 CSS 中：

- `.chat-user-msg` 的圆角 `rounded-[18px_18px_4px_18px]` 是聊天气泡特有的非对称圆角，不应存在于全局 CSS
- `.preview-side`、`.preview-main`、`.preview-toolbar` 是文档预览页的布局类，与设计令牌无关
- `.ws-seg`、`.ws-team-label`、`.ws-fullname-btn` 是工作区切换器的内部样式，应封装在组件内

**影响**：CSS 与组件分离导致「改一个组件需要跨文件搜索 CSS」。新功能开发时倾向于继续在 `index.css` 追加类，形成恶性循环。

#### 问题 5：暗色模式声明但未实现（低）

`tailwind.config.js` 配置了 `darkMode: ["class"]`，但 `:root` 中只定义了亮色变量，`html.dark` 选择器完全缺失。这是一个「承诺了但没兑现」的配置，可能误导开发者以为暗色模式已就绪。

**影响**：用户在系统暗色模式下使用时，半透明背景（`bg-white/55`、`bg-white/85`）会产生不协调的灰白叠加。

#### 问题 6：品牌标记过于简陋（中等）

当前 BrandMark 是一个 `3px × 20px` 的竖条 + `6px × 6px` 的圆点，尺寸仅 `12px × 20px`。在 220px 宽的侧边栏中，它与「知岸」文字的视觉比例严重失衡——标记几乎不可见，无法承载品牌识别功能。

**影响**：品牌存在感极弱。用户在多标签页场景下难以通过 favicon 级别的视觉信号快速定位「知岸」标签页。

## 1.2 交互流程合理性

### 具体问题

#### 问题 1：聊天输入框功能过于简陋（严重）

当前 `ChatInput` 组件是一个单行 `<input>` 元素：

```tsx
<input
  value={value}
  onChange={(event) => setValue(event.target.value)}
  placeholder="提问，答案将附带引用来源…"
  aria-label="向资料库提问"
/>
```

**缺陷清单**：
- ❌ 不支持多行输入（长问题被截断）
- ❌ 不支持 `Enter` 发送 / `Shift+Enter` 换行（当前 `Enter` 隐式通过 form submit 发送，但无视觉反馈）
- ❌ 不支持文件/图片拖拽上传
- ❌ 无输入长度提示
- ❌ 无草稿自动保存（切换线程后输入内容丢失）
- ❌ 无 `@` 提及资料库的快捷选择
- ❌ 无 Markdown / 富文本格式支持

**影响**：作为 RAG 平台的核心交互入口，输入框的能力直接决定了用户提问的深度和质量。当前体验远低于 ChatGPT、Claude 等标杆产品的基线预期。

#### 问题 2：线程管理能力薄弱（中等）

`ThreadListPanel` 的线程列表功能极为基础：

- ❌ 无线程搜索/过滤
- ❌ 无线程重命名（标题由首个问题自动生成，无法修改）
- ❌ 无线程置顶/收藏
- ❌ 无线程文件夹/分组（仅有"今天"/"7天内"/"更早"的时间分组）
- ❌ 删除操作无确认（点击即删，无 undo）
- ❌ 无线程导出

**影响**：活跃用户积累大量对话后，无法有效管理和检索历史对话，导致知识沉淀断裂。

#### 问题 3：无全局快捷键体系（中等）

全站没有任何键盘快捷键：
- ❌ 无 `Cmd/Ctrl + K` 快速搜索 / 命令面板
- ❌ 无 `Cmd/Ctrl + /` 聚焦聊天输入
- ❌ 无 `g d`（go to dashboard）类 Vim 导航
- ❌ 无 `?` 查看快捷键帮助
- ❌ 无 `Esc` 关闭模态的统一处理（各组件各自实现）

**影响**：效率型用户（开发者、研究员）无法形成肌肉记忆，每次操作都需要鼠标点击。这在「知识工作平台」品类中是显著的体验短板。

#### 问题 4：加载状态缺乏进度感知（中等）

- 文档上传/处理：仅显示 `animate-pulse` 的骨架屏和「整理中」状态标签，但无进度条或预计时间
- RAG 检索：流式输出有光标动画（`.chat-stream-cursor`），但首次响应前的等待无感知（无"正在检索..."状态）
- 页面切换：无路由级加载指示器（顶部进度条）

**影响**：RAG 检索通常需要 1-3 秒，用户在这段时间内不知道系统是否在工作，容易重复点击或认为卡死。

#### 问题 5：Toast 通知系统局限（低）

当前 Toast 是页面级组件（各页面自行渲染 `<Toast />`），而非全局 Provider：

```tsx
// 每个 Page 内部
const { toast, show, dismiss } = useToast();
// ...
<Toast message={toast?.message} onDismiss={dismiss} />
```

**影响**：
- 跨页面操作无法显示持续通知（如「文档上传完成」通知在上传后跳转页面时丢失）
- 多个 Toast 无法堆叠
- 无 Toast 类型区分（success / error / warning / info）

#### 问题 6：破坏性操作缺乏保护（中等）

- 删除线程：直接删除，无确认弹窗，无 undo
- 删除资料库：有确认弹窗（`DeleteKnowledgeBaseDialog`），但仅要求输入名称确认，过于繁琐
- 删除文档：有确认弹窗（`DeleteDocumentDialog`），体验尚可
- 退出团队：有确认表单，但无 grace period（立即生效）

**影响**：删除线程的体验最差——用户误触删除按钮后无法恢复对话历史。

## 1.3 信息架构清晰度

### 具体问题

#### 问题 1：侧边栏导航扁平且无图标（严重）

当前侧边栏导航结构：

```
导航
  概览
  资料库
  对话
  组织与部门    ← 仅管理员可见
  成员管理      ← 仅管理员可见
  团队设置      ← 仅管理员可见
  操作审计      ← 仅管理员可见
──────────────
  账号设置
  [用户信息块]
```

**缺陷**：
- ❌ 无导航图标——纯文字列表，视觉区分度极低，快速扫描困难
- ❌ 无分组分区——核心功能（概览/资料库/对话）与管理功能（组织/成员/审计）混在同一列表
- ❌ 管理员视角下有 7 个导航项，但无折叠/分区
- ❌ 无「最近使用」或「收藏」快捷入口
- ❌ 无导航项 badge（如未读消息数、处理中文档数）

**影响**：随着功能增长，侧边栏会越来越拥挤。管理员用户需要从 7 个文字项中找到目标，认知负荷高。缺乏图标导致视觉扫描效率低于竞品（Notion、Slack 均采用图标+文字双编码）。

#### 问题 2：面包屑功能薄弱（中等）

当前面包屑在 `AppTopbar` 中渲染，但仅为静态文本：

```tsx
// 路由 handle 定义
{ breadcrumb: <>资料库 / <b>详情</b>> }
```

**缺陷**：
- ❌ 面包屑不可点击——无法通过面包屑向上导航
- ❌ 不包含动态信息——如「资料库 / 产品文档库 / 详情」中「产品文档库」是静态的「详情」
- ❌ 无 truncation——长名称会撑爆 topbar

**影响**：面包屑失去了导航功能，沦为页面标题的替代品。用户在深层级页面（如文档预览：资料库 / 文档 / 预览）无法快速返回上级。

#### 问题 3：设置入口分散（中等）

设置功能分布在三个独立页面：
- `/settings/account` — 账号设置（密码修改、个人资料）
- `/organization/settings` — 团队设置（团队名称、所有权转移）
- `/organization/members` — 成员管理

这些页面之间无 tab 切换或侧边导航，用户需要通过侧边栏分别进入。

**影响**：用户在「修改密码」后想「查看团队成员」需要两次导航跳转。对比 GitHub Settings（左侧 tab 列表 + 右侧内容），体验差距明显。

#### 问题 4：Dashboard 信息密度过低（中等）

当前 Dashboard 内容：
1. 文档搜索条
2. 主标题区 + 快速提问
3. 状态横幅（条件渲染）
4. 4 列统计卡片（资料库数 / 已上传文件 / 可提问文件 / 7日提问）
5. 运营指标
6. RAG 指标

**缺陷**：
- ❌ 无最近活动流（recent activity feed）
- ❌ 无「最近对话」快捷入口
- ❌ 无「处理中文档」实时进度
- ❌ 无可定制 widget / 拖拽布局
- ❌ 无时间范围筛选器（固定 7 日）

**影响**：Dashboard 是用户进入应用的第一屏，但当前仅提供静态统计数字，缺乏「下一步该做什么」的行动引导。

#### 问题 5：无全局搜索 / 命令面板（严重）

全站无任何形式的跨内容搜索：
- ❌ 不能搜索资料库
- ❌ 不能搜索对话历史
- ❌ 不能搜索文档内容
- ❌ 不能快速跳转到设置项

**影响**：在知识管理平台中，搜索是核心功能。当前用户只能通过侧边栏逐级浏览找到内容，效率极低。标杆产品（Notion Cmd+K、Linear Cmd+K、Slack Cmd+K）均已将命令面板作为核心交互。

## 1.4 响应式适配

### 现状

项目使用 **单一断点** `max-width: 768px`，在此之上为桌面端，之下为移动端。

### 具体问题

#### 问题 1：缺失平板断点（中等）

768px ~ 1024px 的平板区间无专门适配：
- 侧边栏固定 220px，在 768px ~ 900px 的窄屏上主内容区过窄
- 资料库卡片在 `sm:grid-cols-2`（640px+）即变为双列，但 768px~1024px 仍为双列，未利用宽屏空间
- 统计卡片在 `md:grid-cols-4`（768px+）变为四列，但在 768px~900px 四列过于拥挤

**影响**：iPad 用户（768px / 810px / 1024px）体验介于桌面和移动之间，布局比例不协调。

#### 问题 2：大屏未利用（低）

1440px+ 的大屏显示器上：
- 主内容区无 `max-width` 限制，聊天内容 `max-w-[680px]` 居中（合理），但 Dashboard 统计卡片四列等宽铺满，在大屏上卡片过宽
- 侧边栏固定 220px，大屏上可以更宽或支持展开/收起

**影响**：大屏空间未被有效利用，但也未造成体验问题。优先级较低。

#### 问题 3：移动端聊天体验受限（中等）

移动端（≤768px）聊天页面：
- 线程列表默认隐藏，需点击「历史」按钮打开抽屉
- 线程列表抽屉宽度 260px，在 375px 屏幕上占 69% 宽度——过宽
- 输入框区域 `.chat-input-wrap` 的 `px-6`（24px）在小屏上浪费空间
- Agent 模式切换栏 `.agent-mode-bar` 在小屏上可能换行，挤压消息区

**影响**：移动端用户（可能是通勤场景的碎片化使用）每次查看历史对话都需要额外操作，且抽屉过宽影响消息区可见性。

#### 问题 4：无响应式排版缩放（低）

全站使用固定 rem 值，无 `clamp()` 或媒体查询字号缩放。在移动端，标题和正文的字号比例与桌面端完全相同，缺乏移动端的视觉层次调整。

**影响**：移动端信息密度偏高，但不影响可用性。

#### 问题 5：移动端侧边栏抽屉缺少焦点陷阱（中等）

移动端侧边栏通过 `transform: translateX` 滑入，但：
- ❌ 无 focus trap——Tab 键焦点会跑到侧边栏背后的页面内容
- ❌ 打开时未自动聚焦侧边栏首个导航项
- ❌ 屏幕阅读器可能不知道侧边栏已打开（遮罩 `aria-hidden` 设置了，但侧边栏本身无 `aria-modal` 或 `role="dialog"`）

**影响**：键盘用户和屏幕阅读器用户在移动端无法有效使用侧边栏导航。

## 1.5 可访问性 Accessibility

### 现状

项目在可访问性方面已有一定基础——ARIA 属性覆盖了主要交互元素（`aria-expanded`、`aria-controls`、`role="alert"`、`role="tablist"` 等）。但存在多处关键缺口。

### 具体问题

#### 问题 1：无 Skip-to-Content 链接（严重）

全站无 `<a href="#main" class="skip-link">跳到主内容</a>` 元素。键盘用户每次页面切换都需要 Tab 遍历整个侧边栏才能到达主内容区。

**影响**：违反 WCAG 2.1 SC 2.4.1（Bypass Blocks）。键盘用户体验严重受损。

#### 问题 2：颜色对比度边界值（中等）

| 前景色 | 背景色 | 对比度 | WCAG AA（正常文字） | WCAG AA（大文字） |
|--------|--------|--------|---------------------|-------------------|
| `--mut` `#71717a` | `--bg` `#fafaf8` | ~4.7:1 | ✅ 勉强通过 | ✅ |
| `--mut` `#71717a` | `--surf` `#ffffff` | ~4.4:1 | ❌ 略低于 4.5:1 | ✅ |
| `--auth-muted` `#7a6e6a` | `--auth-card` `#fffcfa` | ~4.2:1 | ❌ 不通过 | ✅ |
| `--status-ok-text` `#4a5d47` | `--status-ok-bg` `#e5ebe3` | ~5.8:1 | ✅ | ✅ |
| `--action` `#cb6b3d` | `#ffffff`（按钮文字） | ~3.8:1 | ❌ 不通过 | ✅ |

**重点问题**：赤陶橙 `#cb6b3d` 作为按钮背景色配白色文字，对比度仅 3.8:1，不满足 WCAG AA 正常文字标准（4.5:1）。这影响了所有 `brand` 和 `auth` 变体按钮的可读性。

**影响**：低视力用户、强光环境下使用的用户难以辨认按钮文字。

#### 问题 3：流式聊天输出无 live region（严重）

聊天消息通过 SSE 流式输出，但消息容器未使用 `aria-live="assertive"` 或 `aria-live="polite"`。屏幕阅读器用户无法感知新消息到达。

```tsx
// 当前
<div ref={scrollRef} className="chat-scroll">
  <div className="chat-inner">
    {/* 消息列表 - 无 aria-live */}
  </div>
</div>
```

**影响**：视障用户完全无法使用聊天功能——他们不知道 AI 是否在回复、回复了什么内容。

#### 问题 4：表单字段缺少 `aria-required`（中等）

`AuthField` 组件为输入框设置了 `aria-invalid` 和 `aria-describedby`，但未设置 `aria-required="true"`。屏幕阅读器用户无法区分必填和选填字段。

**影响**：视障用户在注册表单中可能跳过必填字段，导致提交失败。

#### 问题 5：动态内容变更无 announcement（中等）

以下场景缺少 `aria-live` 区域通知：
- 文档上传成功 / 失败
- 资料库处理完成
- 线程删除成功
- 工作区切换
- Toast 显示（Toast 有 `role="status"`，但页面切换后 Toast 消失，无法通知）

**影响**：屏幕阅读器用户在关键操作后得不到反馈。

#### 问题 6：焦点可见性不一致（中等）

部分交互元素的 `:focus-visible` 样式缺失或不明显：
- `.thread-list-item`：无 focus 样式（`transition-colors` 但未定义 `focus-visible:ring`）
- `.crumb-link`：仅有 `hover:underline`，无 focus 样式
- `.cite-chip`：无 focus 样式
- `.chat-toolbar-pill`：非交互元素但可能被误认为可点击

**影响**：键盘用户在 Tab 导航时丢失焦点位置，无法判断当前焦点在哪个元素上。

## 1.6 用户情绪体验

### 现状

「暖白 + 赤陶橙」的配色营造了温暖、人文的基调，在视觉第一印象上是舒适的。但深入使用后，情绪体验有多处可改进之处。

### 具体问题

#### 问题 1：空状态缺乏视觉温度（中等）

所有空状态（`KbOnboardingEmptyPanel`、`ChatEmptyPanel`、`KbResultEmptyPanel`）均使用纯文字 + 虚线边框卡片：

```tsx
<div className="kb-result-empty">
  <h3 className="kb-result-empty-title">还没有资料库</h3>
  <p className="kb-result-empty-desc">创建第一个资料库，上传文档后即可对话。</p>
  <div className="kb-result-empty-action">
    <Button>+ 新建资料库</Button>
  </div>
</div>
```

**缺陷**：无插画、无图标、无引导性视觉元素。纯文字的空状态缺乏情绪引导力。

**影响**：新用户首次进入空 Dashboard 时感受冷淡，缺乏「欢迎来到这里，让我们一起开始」的情绪连接。对比 Notion 的空状态（手绘风插画 + 友好文案）、Linear 的空状态（简约图标 + 行动引导），差距明显。

#### 问题 2：无新手引导 / Onboarding（中等）

新用户注册后直接进入 Dashboard，无任何引导：
- ❌ 无功能 tour / 步骤指引
- ❌ 无 Checklist（「上传第一个文档」「发起第一次提问」「查看引用来源」）
- ❌ 无 tooltip 首次提示
- ❌ 无示例资料库 / 示例对话

**影响**：新用户不知道从何开始，可能直接离开。产品的 Time-to-Value（首次价值实现时间）较长。

#### 问题 3：成功反馈缺乏庆祝感（低）

所有成功操作仅通过 Toast 显示一行文字（「资料库已创建」「文档已上传」），无任何视觉庆祝：
- ❌ 无成功图标动画
- ❌ 无 confetti / 微粒子效果
- ❌ 无进度完成的状态转换动画

**影响**：操作成功后的情绪峰值平淡。特别是文档处理完成（一个可能等待数分钟的操作），仅从「整理中」变为「可提问」，缺乏「完成了！」的情绪释放。

#### 问题 4：错误状态缺乏同理心（低）

错误提示统一使用 `AlertBanner`：

```tsx
<div className="alert-warm-err" role="alert">
  {message}
  <button>重试</button>
</div>
```

文案通常为技术性描述（如「加载失败」），缺乏人性化措辞和具体引导。

**影响**：用户遇到错误时感到沮丧且无助，不知道下一步该做什么。

#### 问题 5：品牌人格未充分表达（中等）

「知岸」这个名字富有诗意——「知识的彼岸」，暗示着探索、抵达、成长。但当前 UI 中：
- 品牌标记是抽象的竖条+圆点，与「知岸」的意象无关联
- 无任何品牌叙事元素（如 loading 动画、404 页面、关于页面）
- 文案风格纯功能化，缺乏品牌调性

**影响**：产品缺乏可被记住的「性格」。用户使用后不会产生「这是知岸」的品牌联想，降低了品牌忠诚度。

#### 问题 6：加载体验缺乏品牌化（低）

骨架屏使用通用的 `animate-pulse` 灰色块，无品牌元素。等待期间无品牌化 loading 动画。

**影响**：错失了在等待期间强化品牌印象的机会。

## 2.3 交互动效优化

### 2.3.1 动效设计原则

1. **有目的的动效**：每个动画都服务于信息传达，不为动而动
2. **快速且自然**：过渡时间 150-250ms，缓动函数 `cubic-bezier(0.4, 0, 0.2, 1)`
3. **尊重用户偏好**：`prefers-reduced-motion: reduce` 时禁用所有非必要动画
4. **品牌化**：loading 动画融入品牌元素

### 2.3.2 关键动效方案

#### 页面切换

```
路由切换时：
1. 顶部出现 2px 品牌色进度条（NProgress 风格）
2. 旧内容 150ms 淡出
3. 新内容 200ms 淡入 + 8px 上移
4. 进度条消失
```

#### 聊天消息流式输出

```
AI 回复到达时：
1. 消息块从 8px 下方淡入上移（200ms）
2. 流式文字逐字显示（已有 cursor 动画，保留）
3. 引用 chip 在文字完成后依次淡入（stagger 50ms）
4. 滚动到底部使用 smooth scroll
```

#### 侧边栏导航

```
Hover 态：
1. 背景色 150ms 过渡到 hover 色
2. 左侧出现 2px 品牌色指示条（从 0 高度展开）

Active 态：
1. 背景色即时切换
2. 左侧 3px 指示条从顶部展开到底部（200ms）
3. 文字颜色 150ms 过渡
4. 图标从灰色过渡到品牌色
```

#### 按钮交互

```
Hover：背景色 150ms 过渡 + 1px 上移 + shadow 增强
Active：1px 下移 + shadow 减弱（模拟按压）
Focus：ring 200ms 展开（品牌色 15% 透明度）
Disabled：opacity 150ms 降至 0.5
```

#### 模态/弹窗

```
打开：
1. 遮罩 150ms 淡入
2. 内容 200ms 缩放从 0.95 → 1.0 + 淡入
3. 焦点移至弹窗内首个可交互元素

关闭：
1. 内容 150ms 缩放至 0.98 + 淡出
2. 遮罩 150ms 淡出
3. 焦点恢复至触发元素
```

#### 文档上传进度

```
上传中：
1. 文件卡片淡入
2. 进度条从左到右填充（品牌色渐变）
3. 进度百分比 tabular-nums 实时更新
4. 完成时进度条变为成功色 + checkmark 图标弹入（spring 动画）

处理中（服务端 RAG 处理）：
1. 进度条变为不确定模式（indeterminate，左右循环）
2. 显示「正在解析文档结构…」→「正在生成向量索引…」→「即将完成…」
3. 完成时卡片轻微弹动 + 状态 badge 颜色渐变
```

### 2.3.3 品牌化 Loading

设计一个与品牌相关的 loading 动画——「灯塔」意象（呼应「知岸」的「岸」+ 灯塔指引知识方向）：

```
状态：3 个垂直排列的光点从下到上依次亮起
时长：1.2s 循环
颜色：品牌橙梯度（brand-300 → brand-500）
大小：24px × 32px
```

### 2.3.4 微交互清单

| 场景 | 动效 | 时长 |
|------|------|------|
| 复制成功 | 复制图标 → Checkmark 图标（icon swap） | 200ms |
| 点赞/收藏 | 图标 scale 弹跳 + 颜色变化 | 300ms spring |
| Tab 切换 | 底部指示条滑动 | 200ms |

## 2.5 LOGO 与品牌重塑

### 2.5.1 品牌名评估

**「知岸」** 是一个好名字——「知识的彼岸」意境深远，且中文二字简洁有力。建议保留。

**英文副名**：当前无英文品牌名。建议增加 `Zhian` 作为拼音品牌名（而非意译），保持中文品牌的纯粹性。

### 2.5.2 LOGO 重设计

**当前**：3px 竖条 + 6px 圆点（过于抽象，无法识别）  
**方向**：融合「岸」的意象与「知识灯塔」的概念

#### 方案 A：灯塔光束

```
    ◆          ← 光束顶点（三角形，品牌橙）
   ╱│╲         ← 光束发散
  ╱ │ ╲
 ╱  │  ╲
────┼──┼────   ← 水平线（岸的意象）
    │
    ●          ← 基点（圆点，深色）
```

- 上方三角光束象征「知识指引方向」
- 下方水平线象征「岸/基座」
- 整体形似字母「Z」（知岸拼音首字母）

#### 方案 B：书页与岸

```
  ┌──┐
  │  │          ← 书页/文档意象
  │  │
  ╱~~╲          ← 波浪线（岸与水）
 ╱    ╲
```

- 上方矩形代表知识载体
- 下方波浪线代表岸的边界

#### 方案 C：极简字标

```
  知岸
  ━━     ← 品牌橙横线作为下划线
  Zhian   ← 小号拼音
```

- 纯文字品牌标识
- 橙色下划线是唯一的色彩元素
- 最简洁、最现代

**推荐**：方案 A（灯塔光束），因其在小尺寸下仍可识别，且与品牌名「知岸」形成语义关联。

## 总结

知岸的前端已有不错的基础——CSS 变量体系、暖色设计语言、骨架屏、ARIA 属性覆盖都体现了设计意识。但距离行业领先水准，需要在以下方面系统性提升：

1. **一致性**：从 15+ 种字号、3 种按钮圆角、冗余色彩令牌中建立统一规范
2. **信息架构**：从扁平导航升级为图标+分组+命令面板的多维度导航体系
3. **交互深度**：从单行输入框升级为多行+快捷键+草稿保存的专业输入体验
4. **动效系统**：从无动效升级为有目的、有品牌感的微交互体系
5. **无障碍**：从基础 ARIA 升级为 WCAG AA 全合规
6. **品牌人格**：从抽象标记升级为有故事、有温度的品牌视觉体系

改造的核心原则是**不破坏现有的温暖设计基调，而是在其上注入专业深度感和系统一致性**。
