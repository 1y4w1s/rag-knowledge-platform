## 二、八大水分来源（按严重度排序）

### 🔴 1. CRAG 19% 的实际根因是配置错误（最诚实也最难看）

- **原因**：`bge-large-zh-v1.5`（1280维）配置了 `embedding_dim=512` → 实际 fallback 到 `bge-small-zh-v1.5`（**中文**嵌入模型 512维）
- **后果**：用中文语义空间嵌入英文 query 和文档，向量相似度基本随机
- **影响范围**：所有英文检索场景
- **修复**：换成 `bge-small-en-v1.5`（384维），预期可到 70%+
- **讽刺点**：19% 是目前所有评测中**最诚实的数字**——它反映了系统真正的薄弱环节

### 🔴 2. Enterprise QA "calibration" = 循环验证（56%→98%）

- 原始基线 56% 才是真实水平
- content_contains 字段用 DeepSeek **从文档 chunk 直接提取** → 同一份 chunk 既是答案来源也是检索目标
- 如果在两个不同章节出现相同数值（如 "99.9%"在合同里出现过也在FAQ里出现），检索到错误章节但命中 content_contains → **假阳性**
- 此问题已在 `docs/final-plan-v3.md` 第10行标记

### 🟠 3. Expense QA "calibration" = 修格式不修系统（41%→92%）

- 原始 41% 到宣称 92% 的提升，**系统一行代码没改**
- 全部提升来自修复测试集断言：`1000` → `1,000`（千分位）、`cfo` → `CFO`（大小写）、`30天` → `30 天`（空格）
- 文档原文有格式，但测试断言没对齐 → 修复断言后分数飙升

### 🟠 4. Golden QA 的结构性刷分

| 刷分机制 | 具体数据 | 影响 |
|----------|----------|------|
| 拒答题免费分 | 23/110 题 (20.9%) 只需 no-match 自动命中 | Hit@3 凭空多了约 21% 的"命中" |
| 子串匹配过于宽松 | `"10天"` 在任意chunk中出现即算命中 | 可能命中无关章节 |
| 重复题 | GQ-1 = GQ-9（完全相同的 query 和 expect） | 重复计数刷分 |
| 简单题比例高 | 难度 ≤0.3 的占 47% (52/110) | 大部分是单点提取 |

### 🟡 5. 评分系统性高估（6个bug）

全部来自 `backend/tests/benchmark/runner.py`，详见 [schemas audit](#):

| Bug | 位置 | 影响 |
|-----|------|------|
| correct_rejection_rate 分母用错 | runner.py:283 | 拒答正确率被低估至1/3（但其他指标被高估） |
| recall_at_k 分母固定为1 | runner.py:243 | recall 几乎永远是 100% |
| 失败query静默跳过 | runner.py:165-167 | 检索失败的query不降指标 |
| NDCG 折扣非标准 | runner.py:266 | rank1=rank2折扣，不可对比 |
| 两套引擎匹配逻辑不一致 | runner.py vs loader.py | 同一query可能不同分 |
| 多chunk匹配同一answer | runner.py:226-231 | precision虚高 |

### 🟡 6. Faithfulness 89% 的宽松条件

- **同一 DeepSeek 同时分解和验证** → 自我一致性偏见
- **空答案/无 claims 默认满分** → 生成失败也不扣分
- PARTIAL 按 0.5 分计入→ 明明部分支持也算一半忠实
- 上下文截断 2000 字符 → 长文档后半段信息丢失

### 🟢 7. LLM-as-Judge 的缺失 safeguard

- judge 本身相对严格（高分门槛高），但：
  - 无多 judge 投票集成
  - 无 golden answer 质量校验
  - 无校准集一致性测试
  - 无评分分布监控

### 🔵 8. 真正的问题：benchmark 和实际对话链路脱节

| 环节 | Benchmark | 生产链路 |
|------|-----------|----------|
| 缓存 | 每次重新检索 | 有LRU缓存 |
| 多轮上下文改写 | 无 | 有 contextualize_query |
| 空检索补偿 | 无 | 有 expand/decompose |
| 相关性门控 | 无 | 有 filter_relevant_chunks |
| 自验证 | 无 | 有 verify_answer |

所以 benchmark 测的是"原始检索质量"，用户实际体验可能更差（也可以更好）。

## 五、一句话总结

> **系统对中文单点提取能到 70%（Real Docs 的真实信号），但 README 上的 95%+ 是测试设计问题堆出来的。CRAG 19% 反而是最诚实的数字——如果不是把中文嵌入模型怼到英文数据上，会说中文系统的真实英文检索能力大约在 10% 左右。**
