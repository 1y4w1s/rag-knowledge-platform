# 步骤 3 详细计划：Enterprise QA 测试集修复

> 基于 2026-07-19 全面审查结果
> 问题定位：重复/过短的 content_contains 导致分数虚高

---

## 问题清单

### 问题 A：18 个重复的 content_contains 值

**最严重的**：

| content_contains | 出现次数 | 分布 |
|------|:-------:|------|
| "3" | **5 次** | 产品规格书、FAQ、操作手册…… |
| "30" | **3 次** | 框架合同、季度报告、FAQ…… |
| "专业版" | **3 次** | FAQ、产品规格书…… |
| "企业版" | **3 次** | FAQ、产品规格书…… |
| 长句子 A | **6 次** | 不同章节同一句子 |
| 长句子 B | **5 次** | 不同章节同一句子 |
| 长句子 C | **4 次** | 不同章节同一句子 |

**问题**：同一个 content_contains 值在不同题目中出现。系统检索到任意包含该文本的 chunk（即使来自错误章节）→ 每道匹配的题都算命中 → **7 道题同时虚高**。

### 问题 B：47 个超短 content_contains（<4 字符）

纯数字 "3"、"5"、"30" 在几乎所有文档章节都会出现。检索到错误章节但命中 content_contains → **假阳性**。

### 问题 C：部分 content_contains 含"答："前缀

DeepSeek 生成 content_contains 时用对话语气 "答：免费版0元..."。匹配时检查的是 **chunk.content**（文档原文），不是 DeepSeek 回答。文档原文不会以"答："开头 → **永不命中 → 假阴性**。

### 问题 D：只有 1 道跨文档题

108 题中有 107 题是单文档查询。系统只需检索单个文档即可回答。测试设计未能充分验证跨文档检索能力。

---

## 修复方案

### 修复 A：去重 content_contains

对 18 个重复值，每个保留一个，其余 17 个替换为各自文档章节中的唯一片段。

**原则**：
- 替代 content_contains 长度 ≥ 10 字符
- 替代 content_contains 在该文档中只出现一次
- 替代 content_contains 不应与其他题目的 expect 重叠

### 修复 B：扩展超短 content_contains

47 个 ≤3 字符的 content_contains，每个扩展为包含该关键词的完整短语。

**示例**：

```
改前：content_contains = "3"
改后：content_contains = "合同期限为 3 年"
```

```
改前：content_contains = "30"
改后：content_contains = "提前 30 天书面通知"
```

### 修复 C：去除"答："前缀

找到所有以"答："开头的 content_contains（约 18 个），去掉前缀。

```
改前：content_contains = "答：免费版0元，含5GB存储..."
改后：content_contains = "免费版0元，含5GB存储..."
```

### 修复 D：增加跨文档题（可选）

在现有 108 题基础上，补充 5-10 道需要跨 2 份以上文档才能回答的 L3/L4 题。

---

## 修复原理

当前匹配逻辑会产生假阳性：

```
题目 A（来自操作手册）：content_contains = "3"
题目 B（来自框架合同）：content_contains = "3"
系统检索到合同章节含"3" → 题 A 和题 B 都命中
```

修复后：

```
题目 A（来自操作手册）：content_contains = "第 3 条 巡检安排"
题目 B（来自框架合同）：content_contains = "合同期限为 3 年"
系统检索到合同章节 → 题 A 不命中（唯一性验证通过）
```
